## 1. 产品概述

### 1.1 产品背景

Claude Code 是 Anthropic 提供的命令行工具，帮助开发者使用 Claude 进行编程辅助。随着使用深入，用户面临以下痛点：

- **成本不透明**：不知道每次会话消耗了多少 Token，产生多少成本
- **工具行为未知**：不清楚 Claude 接受了哪些工具调用，拒绝了什么，原因是什么
- **效率无法评估**：无法对比不同项目的 AI 使用效率差异
- **缺乏可视化**：缺乏对 Agent 行为模式的数据化呈现

### 1.2 核心价值主张

Claude Code Analytics 通过监听本地 ~/.claude 目录，实时解析使用数据，提供：

- **实时成本追踪**：每次会话、每个模型、每日/每周/每月的成本可视化
- **工具级效率分析**：精确到每个工具的接受率、拒绝原因、效率对比
- **轻量 DORA 关联**：结合项目目录结构，分析不同模块的 AI 使用效率
- **数据完全本地**：所有数据存储在本地，不上传任何信息

### 1.3 目标用户

| 用户类型 | 核心诉求 | 优先级 |
|---------|---------|--------|
| 深度使用 Claude Code 的开发者 | 了解每次会话消耗，优化使用习惯 | P0 |
| 需要控制 AI 使用成本的团队 | 预算管理、成本分摊、超支预警 | P0 |
| 需要分析 AI 工具效率的工程师 | 工具接受率、效率排名、改进方向 | P1 |
| 技术管理者/Lead | 团队整体效率、项目 ROI 评估 | P2 |

### 1.4 竞争产品分析

#### 1.4.1 现有解决方案

| 产品 | 类型 | 优点 | 缺点 |
|------|------|------|------|
| Claude Code 官方 | 官方CLI | 无需额外工具 | 无数据分析功能 |
| Cursor 内置统计 | IDE插件 | 集成在编辑器内 | 仅限 Cursor 会话，维度单一 |
| 第三方 Token 计算器 | Web工具 | 简单易用 | 只能估算，无法实时追踪 |
| OpenRouter 仪表盘 | Web平台 | 支持多模型聚合 | 不支持本地会话解析 |

#### 1.4.2 差异化机会

Claude Code Analytics 的核心差异化在于：

1. **本地实时解析**——不依赖云端，直接读本地文件，实时更新
2. **工具级分析**——精确到每个工具的接受/拒绝/修改，而不仅是 Token 统计
3. **项目维度关联**——结合文件系统，分析不同项目的 AI 使用效率
4. **DORA 轻量关联**——不需要完整的 DevOps 体系，就能做效率评估

#### 1.4.3 竞争优势

- **数据源独占**：直接读取 Claude Code 本地文件，他人无法复制
- **分析深度**：工具行为分析是其他产品不做或做得浅的维度
- **隐私优先**：数据完全本地，适合对代码隐私敏感的企业用户

## 2. 用户故事与场景

### 2.1 用户故事总览

| 角色 | 故事 | 期望结果 |
|------|------|---------|
| 开发者 A | 作为深度用户，我想看到每次会话的成本，以便调整我的提示词习惯 | 降低单次会话成本 |
| 开发者 B | 作为 Team Lead，我想看到团队各项目的 AI 使用效率，以便发现改进空间 | 提升团队整体效率 |
| 个人用户 C | 作为在意成本的用户，我想设置月度预算上限并收到预警，以免超支 | 控制月度支出 |
| 开发者 D | 作为问题排查者，我想回放某个会话的时间线，了解工具调用的完整过程 | 快速定位问题 |

### 2.2 核心用户场景

#### 场景一：每日成本检查（用户故事：成本透明化）

**用户**：深度开发者，每天使用 Claude Code 2-4小时

**触发条件**：打开应用，查看 Dashboard

**用户行为**：
1. 打开应用，看到今日成本统计卡片
2. 查看成本趋势折线图，判断今日用量是否异常
3. 点击成本异常点，进入该时段会话列表
4. 定位到高成本会话，查看详情

**用户期望**：3秒内了解今日成本概况

#### 场景二：周度效率复盘（用户故事：效率评估）

**用户**：开发者，关注 AI 工具使用效率

**触发条件**：每周五下午，复盘本周工作

**用户行为**：
1. 进入 Agent Behavior 页面
2. 查看本周工具接受率趋势图
3. 查看拒绝原因分布，找出 Top 问题
4. 点击效率低的工具，查看具体案例
5. 调整自己的提示词策略

**用户期望**：能发现 2-3 个具体的改进点

#### 场景三：月度预算管控（用户故事：预算预警）

**用户**：在意成本的个人用户或 Team Lead

**触发条件**：月初设置月度预算，成本接近阈值时收到通知

**用户行为**：
1. 进入设置页面，配置月度预算 100 美元，预警阈值 80%
2. 日常使用时，应用在后台监听数据变化
3. 当累计成本达到 80 美元时，收到系统通知
4. 用户决定是否调整使用频率或预算

**用户期望**：不超预算，或者超支前有预期

#### 场景四：项目效率对比（用户故事：项目维度分析）

**用户**：Team Lead，想评估不同项目的 AI 使用效率

**触发条件**：查看 Projects 页面，对比多个项目

**用户行为**：
1. 进入 Projects 页面，查看项目效率排名
2. 筛选出成本最高和效率最低的项目
3. 点击进入项目详情，查看目录结构分析
4. 发现某大型项目的 tests 目录 AI 使用率异常高
5. 判断是否正常（测试驱动开发）还是需要优化

**用户期望**：能快速定位异常项目

## 3. 产品功能详细设计

### 3.1 Usage Dashboard（使用仪表盘）

**优先级**：P0（Phase 1 必须）

**核心指标卡片**：

| 卡片 | 指标 | 计算方式 |
|------|------|---------|
| 总成本 | 美元金额（保留4位小数） | sum(input_tokens * input_price + output_tokens * output_price) |
| 总Token | 数量（自动单位换算） | sum(input_tokens + output_tokens)，自动换算K/M |
| 会话数 | 整数 | count(distinct session_id) |
| 工具使用率 | 百分比 | sum(tools_accepted) / sum(tools_proposed) * 100% |

**数据刷新策略**：
- **实时监听**：chokidar（或 Rust notify）监听 ~/.claude 目录变化
- **轮询兜底**：每30秒全量扫描一次，防止遗漏
- **稳定窗口**：文件变化后等待100ms再解析，避免读取部分写入数据

---

### 3.2 Session 分析

**优先级**：P0（Phase 1 列表，Phase 2 详情）

**Session 列表筛选字段**：

| 字段 | 筛选项 |
|------|--------|
| 模型 | 全部 / Opus / Sonnet / Haiku |
| 成本范围 | 全部 / <$0.5 / $0.5-2 / $2-5 / >$5 / 自定义 |
| 工具接受率 | 全部 / >80% / 50-80% / <50% |
| 项目 | 全部 / 自动列出检测到的项目目录 |
| 搜索 | 按会话中的关键词搜索 |

**边界条件处理**：
- **会话中断**：显示"数据加载中"，等待文件写入完成
- **超大文件（>10MB）**：提供大文件警告，允许用户选择加载范围
- **文件损坏**：记录错误日志，显示"会话数据不完整"
- **并发写入**：等待100ms稳定窗口再解析

---

### 3.3 成本分析

**优先级**：P0（Phase 1 基础版，Phase 2 完整版）

**预算预警触发逻辑**：

| 触发条件 | 动作 | 通知形式 |
|---------|------|---------|
| 消费达预算的80% | 首次预警 | 系统通知 + 应用内横幅 |
| 消费达预算的95% | 二次预警（警告） | 系统通知 + 应用内横幅 |
| 超出预算 | 超支提醒 | 系统通知 + 应用内横幅 |

> 每个预警级别只触发一次，新周期开始时重置计数器。

---

### 3.4 模型分析

**优先级**：P1（Phase 2）

**内置模型定价**：

| 模型 | Input价格（美元/1M Tokens） | Output价格（美元/1M Tokens） |
|------|---------------------------|--------------------------|
| claude-opus-4 | 15 | 75 |
| claude-sonnet-4 | 3 | 15 |
| claude-haiku | 0.25 | 1.25 |

---

### 3.5 Agent 行为分析

**优先级**：P1（Phase 2）

**核心指标**：

| 指标 | 定义 | 优秀值 | 警告值 |
|------|------|--------|--------|
| 工具响应速度 | 从提出到执行的时间 | <500ms | >2000ms |
| 一次通过率 | 工具一次执行成功的比例 | >90% | <70% |
| 错误重试率 | 需要重试的工具调用比例 | <5% | >15% |

**拒绝原因自动分类**：

| 关键词 | 推断原因 |
|--------|---------|
| security | 安全考虑（Security concern） |
| already done / already exists | 重复操作（Redundant operation） |
| invalid / wrong format | 参数错误（Invalid parameters） |
| not related / outside | 超出范围（Out of scope） |

---

### 3.6 项目分析

**优先级**：P2（Phase 3 完整版）

**项目检测逻辑**：
1. 读取 ~/.claude 目录中的会话记录
2. 从会话记录中提取工作目录（working directory）
3. 将工作目录映射到项目名称（取目录名作为项目名）
4. 支持用户手动关联目录与项目别名

## 5. 非功能需求

### 5.1 性能需求

| 指标 | 要求 | 说明 |
|------|------|------|
| 首屏加载时间 | < 3秒 | Dashboard 首次加载 |
| 数据刷新延迟 | < 1秒 | 文件变化到界面更新 |
| 页面切换响应 | < 500ms | 路由切换体感 |
| 图表渲染时间 | < 1秒 | 1000条数据的图表 |
| 支持最大会话文件 | 50MB | 超过则提示用户 |

### 5.2 兼容性需求

| 项目 | 要求 |
|------|------|
| Claude Code 版本 | 支持 v1.x 和 v2.x 数据格式，需做 schema 版本检测 |
| 操作系统 | Windows 10+（首发），macOS/Linux（后续） |
| 数据目录 | ~/.claude（跨平台自动适配） |

### 5.3 安全需求

| 项目 | 要求 |
|------|------|
| 数据存储 | 完全本地，不上传任何数据到服务器 |
| API Key | 不存储用户 API Key，仅读取会话元数据 |
| 文件访问 | 仅访问 ~/.claude 目录及子目录 |
| CSP | Tauri CSP 限制，前端无法执行任意代码 |

### 5.4 可靠性需求

| 项目 | 要求 |
|------|------|
| 启动检查 | 应用启动时检测 ~/.claude 目录是否存在 |
| 数据完整性 | 文件损坏时记录错误，不阻塞其他功能 |
| 断点续传 | 增量解析支持断点续传，重启后从上次位置继续 |
| 预警可靠性 | 每个预警级别只触发一次，避免重复通知 |

### 5.5 扩展性需求

| 项目 | 要求 |
|------|------|
| 模型定价更新 | model_pricing 表可扩展，支持未来新模型定价 |
| 新增分析维度 | 预留 projects.project_alias 字段，支持多项目关联 |
| 数据导出 | 支持 CSV/JSON 导出，方便二次分析 |

## 6. 成功指标与北极星指标

### 6.1 北极星指标

**"用户周活跃率（Weekly Active Users, WAU）"**

定义：每周至少打开应用1次，并查看至少1个指标的用户比例。

目标：内测期 WAU ≥ 50%（假设10个用户中至少5人每周使用）

> **为什么选 WAU 而不是留存？**
> 这是一个工具型应用，用户不需要每天都打开，但周活跃率能反映"用户觉得它有用"的真实体感。

### 6.2 辅助指标

| 指标 | 定义 | 健康范围 |
|------|------|---------|
| 次日留存率 | 首次使用后次日再次使用的用户比例 | >40% |
| 周人均会话分析次数 | 每周用户平均查看会话详情的次数 | >3次/周 |
| 预算预警触发率 | 设置预算的用户中，当月触发预警的比例 | 20-50% |
| 成本计算准确率 | 用户手动核算成本与应用显示成本一致的比例 | >99% |
| 导出功能使用率 | 使用过导出功能的用户比例 | >20% |

### 6.3 功能优先级参考指标

| 功能 | 用到才算成功 | 说明 |
|------|-------------|------|
| Dashboard | 80% 用户每周查看 ≥1次 | 核心入口，使用率必须高 |
| Session 详情 | 50% 用户每周查看 ≥1次 | 深度分析功能 |
| 预算预警 | 30% 用户设置预算并触发 ≥1次 | 增值功能 |
| 项目分析 | 20% 用户每周使用 | 高级功能，预期使用率低 |
| 数据导出 | 10% 用户每月使用 | 导出为少数人需求 |

## 7. 风险评估与应对

### 7.1 风险清单

| 风险 | 概率 | 影响 | 应对策略 |
|------|------|------|---------|
| **Claude Code 数据格式变更** | 中 | 高 | PRD 中明确要求做 schema 版本检测；建立版本更新通知机制 |
| **文件解析性能不足** | 低 | 中 | Phase 1 优先验证大文件（>10MB）场景；增量解析优化 |
| **用户隐私顾虑** | 中 | 高 | 强调数据完全本地；不开通任何云端同步功能 |
| **竞争对手复制** | 低 | 中 | 核心差异化在工具行为分析，需持续深耕护城河 |
| **macOS/Linux 支持复杂度** | 中 | 低 | Phase 1 专注 Windows；跨平台路径适配在 Phase 2 考虑 |
| **Token 计算不准确** | 低 | 高 | 模型定价内置验算公式；提供用户手动校正入口 |

### 7.2 数据准确性保障

> ⚠️ **核心假设**：Claude Code 官方未公开完整数据存储 API。本 PRD 基于以下假设构建，在 Phase 1 开发前，必须通过逆向工程验证假设的正确性。

**必须验证的假设清单**：

| 假设 | 验证方法 | 如果假设错误怎么办 |
|------|---------|-----------------|
| metadata.json 包含 cost 字段 | 实际读取多个会话文件，验算成本 | 改用 conversation.jsonl 自行计算 |
| conversation.jsonl 记录完整 tool_calls 生命周期 | 检查是否有 accepted/rejected 状态 | 改为监听 Claude Code 日志文件 |
| 文件格式在版本间稳定 | 读取不同版本 Claude Code 的文件 | 添加版本检测逻辑，多版本兼容 |

## 9. 技术架构

**核心技术栈**：

| 层级 | 技术选型 | 选型理由 |
|------|---------|---------|
| 桌面框架 | Tauri 2.x | 比 Electron 轻量（打包体积小），Rust 后端性能强 |
| 前端框架 | React + TypeScript | 生态丰富，组件库完善 |
| 状态管理 | Zustand | 轻量，适用 Tauri 场景 |
| 图表库 | ECharts | 功能强大，支持定制 |
| 数据库 | SQLite（rusqlite） | 轻量，无需单独安装服务 |
| 文件监听 | Rust notify crate | 纯 Rust 实现，无需 Node.js 依赖 |

**Tauri IPC 通信模式**：
- 前端通过 `invoke()` 调用 Rust 命令
- Rust 通过 `events` 主动推送事件给前端
- 文件变化事件 → Rust 解析 → 推送 `data_updated` 事件 → 前端刷新 UI

## 10. MVP 分阶段开发计划

### Phase 1：核心数据收集与展示（2-3周）

**目标**：能够运行应用，基本数据可见

**P0 功能**：
- [ ] Dashboard（统计卡片 + 成本趋势图 + 最近会话列表）
- [ ] Sessions 列表（时间筛选 + 分页）
- [ ] 数据采集（监听目录 + 解析 metadata.json + 解析 conversation.jsonl 基础字段）
- [ ] SQLite 数据持久化

**验收标准**：
- 应用可以启动并显示 Dashboard
- 解析 ~/.claude 目录下的会话数据
- 成本计算正确（与手动核算误差 < 0.001）
- 数据重启后不丢失

---

### Phase 2：完整功能实现（3-4周）

**目标**：所有核心功能完整可用

**P0 功能**：
- [ ] Session 详情（概览面板 + 时间线视图 + 工具调用详情）
- [ ] Cost Analysis 完整（多 Tab + 成本明细表）
- [ ] 预算预警（系统通知 + 应用内横幅）
- [ ] Model Analysis（多模型对比卡片 + 雷达图）
- [ ] Agent Behavior（接受率统计 + 拒绝原因分析 + 工具效率排名）

**P1 功能**：
- [ ] Projects 基础版（项目检测 + 概览卡片）

---

### Phase 3：高级分析与体验优化（2-3周）

**目标**：差异化功能，优秀用户体验

## 12. 待评审问题

> 以下问题需要在评审会议上确认：

1. **数据源依赖风险**：Claude Code 未公开 API，数据格式可能在版本更新时变化。是否接受这个风险，并据此调整 Phase 1 验收标准？

2. **MVP 范围确认**：Phase 1 是否只做 Dashboard + Sessions 列表即可交付，还是需要包含更多功能才算 MVP？

3. **平台优先级**：是否确认 Windows 首发，macOS/Linux 在 Phase 2 考虑？

4. **成功指标确认**：WAU 作为北极星指标是否被认可？目标值（≥50%）是否合理？

5. **竞争策略**：工具行为分析作为核心差异化，是否需要在 PRD 中更突出强调？
