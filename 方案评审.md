# Claude Code Analytics 方案评审

## 一、技术可行性结论

### 1.1 目录监听方案

**可靠性分析：**

chokidar 监听 `~/.claude` 是可行的，但存在以下技术细节需要注意：

| 风险点 | 说明 | 应对 |
|--------|------|------|
| Windows 路径 | `~/.claude` 在 Windows 上解析为 `C:\Users\<user>\.claude` | 使用 `os.homedir()` 动态构建路径 |
| 文件格式 | 需要确认 `.claude` 内的 JSON 结构 | 先做 schema 逆向工程 |
| 权限问题 | Windows 文件锁、UAC 权限 | 降级处理，监听失败时提示用户 |
| 大量文件 | Sessions 目录可能膨胀 | 增量监听 + 懒加载 |

**核心问题：** `.claude` 目录里到底有什么文件？Session 数据存在哪个路径？这一步不明确，监听方案是空中楼阁。

### 1.2 React + Vite 选型

**评估：✅ 合适**

- 轻量级 SPA，数据可视化用 Recharts/Echarts
- Vite HMR 在开发阶段体验好
- 纯前端无后端，构建产物可本地也可部署

**技术坑：**

1. **Electron vs 纯浏览器**：纯浏览器无法直接访问本地文件系统。需要明确 —— 是打包成 Electron 应用（访问本地），还是 Web 版（用户手动上传/导入数据）？
2. **文件监听**：浏览器环境不能直接用 chokidar，必须 Electron 或 Tauri
3. **数据更新**：监听变化后如何通知 UI —— 轮询、WebSocket、还是 IPC？

### 1.3 关键技术依赖

```
// 必须确认的依赖
- chokidar (Electron main process)
- electron / tauri (桌面打包)
- recharts 或 echarts (图表)
- electron-store 或 similar (本地配置)
```

## 二、产品可行性结论

### 2.1 功能实现可行性

| 功能 | 实现难度 | 前提条件 |
|------|----------|----------|
| Usage Dashboard | ⭐⭐ 低 | 数据聚合 |
| Session 分析 | ⭐⭐⭐ 中 | 需要 session 文件格式 |
| 成本排行 | ⭐⭐ 低 | session 有 cost 字段 |
| 模型分析 | ⭐⭐ 低 | session 有 model 字段 |
| Agent 行为统计 | ⭐⭐⭐ 中 | 需要理解 tool use 日志结构 |
| 项目分析 | ⭐⭐⭐ 中 | 需要理解 project 维度如何定义 |

**关键前置工作：** 你需要逆向 `.claude` 目录结构。建议第一步就做这件事。

### 2.2 价值验证优先级

MVP 最能验证价值的2个功能：

1. **成本排行** — 用户最直接的痛点是"这个 Session 花了多少钱"，排行是强需求
2. **Usage Dashboard** — 一眼看全局，证明数据可采集

### 2.3 用户授权意愿

**分析：**

- 磁盘权限是**高敏感权限**，用户会警惕
- 但如果是**桌面应用**，用户授权心理门槛相对低
- 如果是**浏览器 Web**，需要解释为什么需要这权限

**建议：** 采用 Electron/Tauri 桌面应用形态，比浏览器版更容易获取信任。

## 三、Top 3 风险 + 应对方案

### 风险 1：数据源不稳定（致命）

**描述：** Claude Code 的内部数据结构（session 格式、字段名）没有公开 API，随时可能因版本更新而破坏解析逻辑。

**应对：**
- 明确声明"基于当前版本 X.Y.Z 测试"
- 实现 schema 版本检测，版本不匹配时提示用户
- 预留数据迁移机制

### 风险 2：用户隐私顾虑（高）

**描述：** 监听用户 home 目录是极高敏感操作，用户会质疑"你看了我什么数据"。

**应对：**
- 所有数据**本地处理**，不上报任何服务器
- 明确公示数据处理逻辑（代码开源或白皮书说明）
- 提供"只读模式"：用户手动导入特定 session 文件，而非全目录监听

### 风险 3：Electron 打包复杂度（中等）

**描述：** Electron 开发、签名、打包 Windows 安装包有学习曲线，且安装包体积大（~100MB+）。

**应对：**
- 考虑 Tauri 替代（Rust 后端，体积小 10x）
- 使用 electron-builder / tauri bundler 简化打包
- 早期验证阶段可用"导入 JSON 文件"模式绕过打包

## 四、推荐优先级

### MVP 实施排序

| 优先级 | 功能 | 理由 |
|--------|------|------|
| **P0** | Usage Dashboard | 展示数据可采集，证明方案可行 |
| **P0** | 成本排行 | 用户最强痛点，直接支撑产品价值 |
| **P1** | Session 分析 | 详情页，增强粘性 |
| **P2** | 模型分析 | 实现简单，可复用 Dashboard 基础设施 |
| **P3** | Agent 行为统计 | 需要额外数据源（tool use log），MVP 后做 |
| **P3** | 项目分析 | 需要定义 project 维度，最后做 |

### 分阶段建议

**Phase 1（MVP，2-3周）：**
- 实现"导入 session JSON 文件"导入模式（绕过目录监听）
- Usage Dashboard + 成本排行
- Electron 桌面壳

**Phase 2（完整版，2-3周）：**
- 增加目录监听能力
- Session 详情页
- 模型分析

**Phase 3（可选，长期）：**
- Agent 行为统计
- 项目分析
- 数据导出/报告功能

## 五、关键问题清单

在进入开发前，你必须回答：

1. ✅ `.claude` 目录结构是什么？Session 文件路径和格式？
2. ✅ Session 数据中 cost 字段存在吗？格式是什么？
3. ✅ Electron（还是 Tauri）？这决定了数据监听方案
4. ✅ 数据是否完全本地处理？有没有上报计划？

---

**总结：** 技术上可行，但核心依赖对 `.claude` 数据结构的逆向。建议先做 schema 调研，再决定技术路径。P0 功能（Dashboard + 成本排行）可以独立验证价值，建议从"导入 JSON 文件"模式开始，降低初期开发复杂度。
