### 2.2 各产品详细功能

#### GitHub Copilot Analytics（GA 2026年2月）

**核心指标：**
- Adoption 类：DAU、WAU、MAU、Copilot 激活率
- Engagement 类：acceptance rate、user_initiated_interaction_count、code_acceptance_activity_count
- Code Generation 类：suggested_lines、accepted_lines、rejected_lines、deleted_lines
- PR Lifecycle 类：PR 创建到合并时间

**Dashboard 层级：**
- 企业级聚合视图（Enterprise → AI Controls → Copilot → Metrics）
- 组织级视图（Insights → Copilot）
- 支持按编辑器、编程语言、模型维度下钻

**数据源：** IDE Telemetry（Tzap、Copilot Plugin）+ GitHub 事件关联

**访问要求：** Enterprise Owners / Org Admins / Billing Managers

---

#### Cursor Stats

**Dashboard 类型：**
- Admin Dashboard（cursor.com）：包含请求数、用量支出、关键活动指标
- Teams Plan：Usage stats + reporting
- Enterprise：AI Code Tracking API + Audit Logs + 更细粒度控制

**导出方式：**
| 方式 | 内容 | Plan 要求 |
|------|------|----------|
| CSV Export | 每用户每日指标：suggestions shown/accepted、lines added/deleted、premium requests、active status | Teams+ |
| Admin API | 成员信息、消费数据、按模型/token 类型拆分的详细事件 | Enterprise |
| Analytics API | 每日代码编辑/AI协助/活跃用户指标 | Enterprise |

**数据采集：** 编辑器端直接收集，事件发送至 Cursor 后端

---

#### JetBrains AI Analytics（2026年2月新发布）

**产品名：** JetBrains Console — AI Management & Analytics

**功能模块：**
1. **Active AI Users** — 追踪活跃 AI 用户趋势
2. **AI Credit Consumption** — 追踪 Credit 消耗（基础+Top-up）
3. **Credit Limit Reached** — 识别频繁触达上限的用户
4. **AI-Generated Code & Acceptance Rate** — 代码接受率 + 工具间横向对比
5. **AI-Modified Code** — 不同 AI 工具在代码库中的占比
6. **AI Feature Activity** — Chat 使用量、建议量

**工具覆盖：** Junie、Claude Code（第三方）、OpenAI Codex、JetBrains AI Assistant

**定价：** 包含在商业 AI 许可中，无需额外付费

**访问：** JetBrains Console，角色权限可配置

---

#### Anthropic Claude Code Analytics（原生）

**来源：** Claude Console（Analytics → Claude Code）

**适合对象：** Team Plan Owners/Primary Owners / Enterprise Owners/Admins / API Console Admin

**指标（Organization 级）：**
- Lines of code accepted（团队接受的 AI 建议代码行数）
- Suggestion accept rate（建议接受率）
- Activity trends（每日活跃用户 + Sessions 趋势）
- Lines accepted over time（每日接受代码行趋势）

**指标（Contribution 级，需 GitHub App）：**
- PRs merged（有无 Claude Code 协助的合并 PR 数）
- Lines of code committed（有无 Claude Code 协助的提交行数）
- PRs opened per user（用户维度 PR 活动）

**数据关联方式：** Claude Code Session Activity ↔ GitHub Commits & PRs 关联

**限制：**
- 数据按月重置，每月一个周期
- 每日处理，非实时（允许 24h 延迟）
- **不支持 Bedrock/Vertex 用户**（第三方端点数据不进入 Console）
- **无 Token 级消费明细**

---

#### Claude Usage Analytics（第三方 VS Code 插件）

**作者：** Reid Havens / Analytic Endeavors

**定位：** 个人开发者视角，VS Code 内实时快览

**功能：**
- 7个状态栏 Widget：总花费、今日花费、消息数、Token数、缓存效率、Subscription 级别
- 4-Tab Dashboard：Overview / Cost / Messages / Personality
- 成就徽章系统（Coding streaks）
- 外部数据集成（external-additions.json，支持接入 Copilot CLI 等其他工具数据）

**定价：** 免费（VS Code Marketplace）

---

#### Raycast AI Stats（社区 Extension）

**定位：** AI 模型性能排行榜 + 价格/吞吐量对比

**功能：**
- 模型搜索（按名称/创作者）
- 价格标签（$/1M Token 输入/输出）
- 吞吐量指标（Median tokens/sec、TTFT）
- 排行榜（MMLU、GPQA、TPS 等 benchmark）

**数据源：** ArtificialAnalysis.ai API

**注意：** 这**不是** Raycast 使用分析，而是 AI 模型市场数据快览

---

#### Jellyfish（第三方工程智能平台）

**定位：** 将 AI 工具指标与工程交付结果关联

**AI 相关功能：**
- Copilot 专用 Dashboard（接受率、每日活跃趋势）
- 关联 Copilot 使用 → PR velocity / Cycle time / Change failure rate
- 自定义基准测试（按 Claude Code adoption level 对比团队）
- 支持工具：Copilot、Cursor、Gemini Code Assist、Sourcegraph 等

**数据连接：** OpenTelemetry export → Prometheus/Grafana（可选实时）

**Jellyfish 指出的 Claude Code 原生分析缺陷：**
1. 不支持 Bedrock/Vertex 端点数据（多端点用户数据碎片化）
2. 无实时 API 数据（不能接入 Prometheus/Grafana）
3. 使用与交付结果未关联（不知道 AI 使用是否真的在加速交付）

## 八、可借鉴的设计

### 8.1 必选借鉴（Claude Code 短板）

1. **实时成本追踪（JetBrains 模式）**
   - 用户最痛点：Claude Code 无成本透明度
   - 借鉴：状态栏实时 cost widget + 预警阈值 + 多人使用聚合

2. **工具级接受率细分（JetBrains 模式）**
   - 原生只有整体 accept rate，无工具维度
   - 借鉴：按 `/edit`、`/notebook-edit`、`/bash`、`/web` 拆分接受率

3. **DORA 指标关联（Jellyfish 模式）**
   - AI 使用 → PR velocity / Cycle time / Change failure rate
   - 这是管理层最关心的：AI 工具到底有没有加速交付

4. **Credit Limit 预警（JetBrains 模式）**
   - 帮助团队 Admin 预判上限触达，避免工作中断

### 8.2 可选借鉴

5. **个人 Gamification（Claude Usage Analytics 插件模式）**
   - Coding streaks + 成就徽章 → 提升个人用户黏性

6. **外部数据集成接口（Claude Usage Analytics 插件模式）**
   - 提供 JSON 文件接入规范，用户可合并 Copilot CLI 等其他数据

7. **多 Agent 横向对比（JetBrains 模式）**
   - 当 Claude Code + Copilot + Codex 并存时，帮助团队评估工具效果

## 九、差异化机会点

基于以上分析，**Claude Code Analytics** 的差异化机会在于：

### 机会 1：成本实时可见 + 预算预警
> 竞品全缺。Claude Code 用户最大痛点是没有成本透明度。

- 实时 Token 消耗 + 换算成美元
- 预算 pace 追踪（"按当前速度本月将花费 $X"）
- 多用户聚合视图（Admin）
- 预警阈值配置（超额前通知）

### 机会 2：工具级使用效率分析
> JetBrains 做了但 Claude Code 没有；Claude Code 用户行为高度依赖 `/tools`

- `/edit` vs `/write` vs `/notebook-edit` vs `/bash` vs MCP tools 接受率
- Session 内 Tool 调用频次分布
- Subagent 使用模式分析（深度用户特征）

### 机会 3：交付结果闭环
> Jellyfish 独家，但太贵；Claude Code 有 GitHub 天然数据优势

- AI 协助 PR vs 非 AI PR 的 Cycle Time 对比
- AI 使用强度（按用户/团队）↔ 交付指标相关性
- 新人 onboarding 时间缩短估算

### 机会 4：跨云端点聚合
> Copilot 原生也没有，但 Bedrock/Vertex 用户有强烈需求

- 统一 Dashboard 同时覆盖 Anthropic 原生 + AWS Bedrock + GCP Vertex 端点
- 解决多端点用户数据碎片化问题

### 机会 5：企业级 Export + BI 集成
> JetBrains 无 API，Cursor 只有 Enterprise 有

- 提供 NDJSON/CSV Export（Teams Plan 起）
- REST API 支持（Enterprise）
- 兼容 Grafana/Prometheus（OpenTelemetry）
- 这三点竞品均未完整提供

## 附录 B：PRD 评审意见（问题清单）

### B.1 优先级 P0（需明确，否则无法开发）

| # | 问题 | 影响 | 建议 |
|---|------|------|------|
| B1 | **Claude Code 数据结构未验证** | PRD 假设 `metadata.json` 包含 `total_cost`、`input_tokens`、`output_tokens` 字段，`conversation.jsonl` 包含完整的 tool_calls 生命周期，但竞品分析和方案评审都未确认这些假设 | 开发前必须先做 schema 逆向，明确数据源实际字段。Phase 1 MVP 应以"手动导入 JSON 文件"模式替代目录监听，降低依赖风险 |
| B2 | **跨平台路径处理缺失** | Windows 路径为 `C:\Users\<user>\.claude`，macOS 为 `/Users/<user>/.claude`，Linux 为 `/home/<user>/.claude`。PRD 只提 `~/.claude`，未说明跨平台适配方案 | 在"技术架构"章节增加跨平台路径处理说明，使用 Rust 的 `dirs` crate 或 Tauri 的 `PathResolver` |
| B3 | **预算预警的会话跨周期问题** | 预警逻辑中"新周期开始时重置计数器"，但如果一个会话跨越多个计费周期（如月末开始、月初结束），成本如何分配？ | 补充说明：按会话结束时间所在周期计算成本归属 |
| B4 | **大文件边界条件未定义** | PRD 说"超过10MB提供大文件警告，允许用户选择只加载头部/尾部/全量"，但未定义 UI 交互细节 | 补充：提供"快速预览"模式（仅解析前100行）和"完整加载"选项 |
| B5 | **数据 Schema 版本兼容性** | Claude Code 版本更新可能导致数据结构变化，PRD 未说明版本检测和兼容性策略 | 增加"版本检测"机制，版本不匹配时提示用户并暂停监听 |

| B8 | **成本计算依赖的数据字段未验证** | PRD 假设 `metadata.json` 中存在 `input_tokens` 和 `output_tokens`，但竞品分析显示 Claude Code 原生"无 Token 级消费明细" | 明确成本数据来源：如果 metadata.json 没有，需要从 conversation.jsonl 逐条累加，或提示用户成本数据可能不准确 |

| B10 | **工具拒绝原因自动分类的准确性问题** | 2.5.5 说"通过关键词匹配推断"拒绝原因，但关键词覆盖度、误判率未说明 | 增加免责声明：该分类基于启发式规则，可能不准确；提供用户手动修正入口 |
